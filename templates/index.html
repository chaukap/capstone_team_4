About SDP

Privacy made simple and easy.

Every business generates insightful data. Medical data can improve the quality of life for patients. Human resource data can inform a business on how best to satisfy their employees. All data, in the right hands, can benefit humanity.

However, most health and human resource data is amassed in cloud storage and only accessed sparingly for fear of privacy breaches. Our solution empowers data owners to share their valuable information while protecting the customers that trust them.

Differential privacy is the technique we use to mask the existence of any single individual in a dataset. By hiding the existence of an individual, we can protect their privacy. The mathematics behind differential privacy is intimidating, but our solution simplifies the process to a few simple decisions so you can start earning passive income from your data instantly.

    • The Laplacian Mechanism is focused on numerical answers. It adds noise directly to the answer itself. What if we want to return a precise answer (i.e. no added noise), but still preserve differential privacy? One solution is the exponential mechanism which allows selecting the “best” element from a set while preserving differential privacy. The analyst defines which element is the “best” by specifying a scoring function that outputs a score for each element in the set, and also defines the set of things to pick from. The mechanism provides differential privacy by approximately maximizing the score of the element it returns - in other words, to satisfy differential privacy, the exponential mechanism sometimes returns an element from the set which does not have the highest score. The biggest practical difference between the exponential mechanism and the Laplace Mechanism mechanisms is that the output of the exponential mechanism is always a member of the set of possible answers. This is extremely useful when selecting an item from a finite set, when a noisy answer would not make sense. For example, we might want to pick a date for a big meeting, which uses each participant’s personal calendar to maximize the number of participants without a conflict, while providing differential privacy for the calendars. Adding noise to a date doesn’t make much sense: it might turn a Friday into a Saturday, and increase the number of conflicts significantly. The exponential mechanism is perfect for problems like this one: it selects a date without noise.
    The parameter epsilon is the parameter that sets the degree of privacy leakage permitted for a particular database. The higher the value of epsilon, the more privacy leakage, the less protection for individuals in the database for identification as members. The SDP platform allows database owner to choose epsilon values between .1 and 5 by increments of .1.
    Practically speaking an epsilon of value 1 or less provides a very high degree of privacy and values of 4 or greater provide little privacy. It is challenging to set the epsilon parameter because there is not a set of hard and fast rules that guide this setting of epsilon for all use cases.
    Setting the epsilon parameter comes with an important trade-off. The lower the value (higher privacy) the lower the utility of the query responses. Since privacy requires a degree of “error” in the query response, setting the parameter requires a balancing of privacy with utility. This balance will depend on particular databases, their population, the sensitivity of their data, and the potential uses of the data. In short, setting the epsilon parameter is more if an art than a science. It may require a degree of experimentation on the part of the database owner.     It is important to understand that differential privacy mechanisms attempt to provide protection against the identification of a particular individual’s membership in a database. The privacy provided is not absolute privacy. The privacy guarantees afforded by differential privacy mechanisms are mathematically rigorous. However, the practical outcomes depend on prior or auxiliary knowledge and the threat model. While the degree of privacy is set by the privacy parameters, the practical implications are use case and threat model specific.

    Broadly speaking sensitivity is the degree to which a query’s outcome can change as it is implemented on two “adjacent” databases. Also broadly speaking an “adjacent” database from a given database, D, is a database D’ that differs by only one record. The sensitivity of a query is the maximum amount a query can differ when considered over ALL of the possible adjacent databases.


    Setting the sensitivity parameter for a Laplacian Mechanism will vary by query:

  • For the COUNT query the sensitivity is always one (or two) as the COUNT in adjacent differs by one record. Setting the sensitivity to 1 sets the “Unbounded” or “local” sensitivity. Setting the sensitivity to 2 will set the “Bounded” or global sensitivity.
  • For SUM the sensitivity should be set by the user depending on the extremes that are possible in the data that you may want to protect. As can be seen in the Jupyter Notebooks Learning Series, protecting the underrepresented requires the database owner to anticipate the range of possible values of the data and. Set the sensitivity to reflect how an additional data point can increase the SUM. See Jupyter Notebook: Laplacian_Mechanisms_Protect_Oski.
  • For the query AVERAGE, the sensitivity can set with similar logic to SUM, but the effect of one additional record on the outcome is a lot smaller than with a SUM query. Setting the sensitivity to reflect the difference between the upper and lower limits of the potential record values (as in SUM) is a safe bet but perhaps overly conservative.
