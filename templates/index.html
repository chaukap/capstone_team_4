About SDP

Privacy Made Simple

Every business generates insightful data. Medical data can improve the quality of life for patients. Human resource data can inform a business on how best to satisfy their employees. All data, in the right hands, can benefit humanity.

However, most health and human resource data, for example, has highly restricted access as a means to prevent privacy breaches. Our solution empowers data owners to share their valuable information while protecting the individuals that trust them with it.

Differential privacy is the technique we use to mask the existence of any single individual in a dataset. By hiding the existence of an individual, we can protect their privacy. The mathematics behind differential privacy is intimidating, but our solution simplifies the process to a few simple decisions so you can start earning passive income from your data instantly.

The Laplacian Mechanism focuses on numerical data only. Noise is added noise to the true numerical answer to a query. So the Laplacian Mechanism returns an answer from the dataset with a degree of error in it. The ExponentialMechanism allows queries of non-numerical data. The mechanism always returns a possible answer from the dataset. This query answer is not always the correct answer. The Exponential Mechanism can also be customized with use case specific utility functions. These utlility functions can specify what constitutes the best answer. The analyst defines which element is the “best” by specifying a scoring function that outputs a score for each element in the set, and also defines the set of things to pick from. The mechanism provides differential privacy by approximately maximizing the score of the element it returns - in other words, to satisfy differential privacy, the exponential mechanism sometimes returns an element from the set which does not have the highest score.

The biggest practical differencebetween the Exponential mechanism and the Laplace Mechanism mechanisms is that the output of the exponential mechanism is always a member of the set of possible answers. This is extremely useful when selecting an item from a finite set, when a noisy answer would not make sense. For example, we might want to pick a date for a big meeting, which uses each participant’s personal calendar to maximize the number of participants without a conflict, while providing differential privacy for the calendars. Adding noise to a date doesn’t make much sense: it might turn a Friday into a Saturday, and increase the number of conflicts significantly. The exponential mechanism is perfect for problems like this one: it selects a date without noise.
The parameter epsilon is the parameter that sets the degree of privacy leakage permitted for a particular database. The higher the value of epsilon, the more privacy leakage, the less protection for individuals in the database for identification as members. The SDP platform allows database owner to choose epsilon values between .1 and 5 by increments of .1.
Practically speaking an epsilon of value 1 or less provides a very high degree of privacy and values of 4 or greater provide little privacy. It is challenging to set the epsilon parameter because there is not a set of hard and fast rules that guide this setting of epsilon for all use cases.
Setting the epsilon parameter comes with an important trade-off. The lower the value (less privacy leakage, higher privacy) the lower the utility of the query responses. Since privacy requires a degree of “error” in the query response, setting the parameter requires a balancing of privacy with utility. This balance will depend on particular databases, their population, the sensitivity of their data, and the potential uses of the data. In short, setting the epsilon parameter is more of an art than a science. It may require a degree of experimentation on the part of the database owner. It is important to understand that differential privacy mechanisms attempt to provide protection against the identification of a particular individual’s membership in a database. The privacy provided is not absolute privacy. The privacy guarantees afforded by differential privacy mechanisms are mathematically rigorous. However, the practical outcomes depend on prior or auxiliary knowledge and the threat model. While the degree of privacy is set by the privacy parameters, the practical implications are specific to your use case and privacy threats.
You cannot prevent attacks but can minimize their effectiveness. By setting the right privacy parameters you can vitiate and thwart an attack to the degree you choose. SDP allows data owners to set an epsilon budget. This budget prevents attackers from repeatedly querying data beyond what data owners believe is safe for individuals.

SDP provides access for users to databases that have been provided by database owners. Access to the databases is enabled through differentially private queries . As a user you can query the database with numerical queries such as COUNT, SUM, AVERAGE and non-numerical data for MOST COMMON, LEAST COMMON, MAXIMUM AND MINIMUM . These aggregation queries enable insights from the data without sharing the individual records.
The trade-off between utility and privacy is maintained by setting two parameters epsilon (privacy leakage parameter) and sensitivity generally referred to as delta (data variability parameter). Setting these parameters is an interactive exercise. Setting these parameters is specific use case. Broadly speaking sensitivity is the degree to which a query’s outcome can change as it is implemented on two “adjacent” databases. Also broadly speaking an “adjacent” database from a given database, D, is a database D’ that differs by only one record. The sensitivity of a query is the maximum amount a query can differ when considered over ALL of the possible adjacent databases.


Setting the sensitivity parameter for a Laplacian Mechanism will vary by query:


  • For the COUNT query the sensitivity is always one as the count in adjacent differs by one record.
  • For SUM, AVERAGE the sensitivity should be set by the user depending on the extreme values that are possible in the data that needs to be protected. To set the sensitivity choose the highest possible value that could be anticipated in the database and enter this value as a upper bound in the query generator. Set the lower bound to the lowest possible value that could be detected in the data. The query generator will calculate the sensitivity for you.

    It is important to remember that differential privacy is not complete privacy. The risk of exposure of an individual's membership in the database is never absolute. By allowing useful information to be obtained through our query engine, a degradation in privacy is inevitable. Perfect privacy can only be achieved with no utility. Perfect utlility affords no privacy. The considerations in a practical balance need to be carefully considered. SDP can track usage for data owners for behaviors associated with attacks, but some level of risk must be assumed.
